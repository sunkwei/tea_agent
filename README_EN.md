# TeaAgent v0.8.2

TeaAgent is a **self-evolving intelligent assistant** built on the OpenAI-compatible Function Calling interface. Core features: **self-extensible toolkit**, **self-evolving system prompts**, **dual-mode personality switching**, and a **three-layer cognitive system** (Memory/Reflection/Subconscious).

Only 10 lightweight core dependencies (openai, numpy, markdown, tkinterweb, pyautogui, mss, Pillow, requests, beautifulsoup4, tkhtmlview). OCR/TTS/ASR are optional extensions. Relies solely on Python's tk library — no browser needed, extremely lightweight. The vast majority of code is generated by the LLM itself; this is an "AI writing AI" experiment project. (Currently using the DeepSeek V4 Pro model for self-evolution — it's cheap!)

## ⚠️ Security Warning

This project has **no security sandbox** — the Agent can execute arbitrary system commands (sudo commands will prompt a GUI password dialog) and modify its own code (use git for version control). It is recommended to run in a virtual machine.

## 🏗️ Architecture Overview

### Auto-Restart (watchdog file monitoring)

When `.py` files outside the `toolkit/` directory change, the GUI **auto-restarts after a 2-second debounce** — no manual action needed:

```
File change → watchdog detection → 2s debounce → _shutting_down gate
    → wait for _sess_lock (max 10s, ensures DB write completes)
    → WAL checkpoint + close DB
    → os.execv() in-place restart
```

> Tool modifications under `toolkit/` go through `toolkit_reload()` hot-reload and do not trigger a restart.

### Three-Level History Loading Strategy

`load_history()` applies a **three-level progressive loading** strategy to conversation history based on recency, balancing context coherence and token savings:

```
conversations (all)
    │
    ├─ Level 3 Summary  ──→ Semantic Summary (user preferences/key conclusions) + Tool-Chain Summary
    │
    ├─ Level 2 Semantic  ──→ Semantically relevant to current task → keep full user+assistant
    │                        Weakly relevant → light summary, irrelevant → discard
    │
    └─ Level 1 Latest Round ──→ user + assistant function_call + tool return + assistant final answer, fully preserved
```

| Level | Content | Strategy |
|-------|---------|----------|
| Level 1 | Latest round | Fully preserved (including function_call / tool results / reasoning_content) |
| Level 2 | Semantically relevant rounds | Relevant → full, weakly relevant → light summary, irrelevant → discard |
| Level 3 | Summary layer | Semantic Summary + Tool-Chain Summary |

### Conversation Flow (single chat_stream)

```
User Input → Pipeline ────────────────────────────────────────→ Output
              │                                                  │
              ├─ ① Memory Injection (inject_memories)            │
              │    CRITICAL instructions injected unconditionally │
              │    Others sorted by relevance×importance×recency  │
              │                                                  │
              ├─ ② Add User Message (add_user_message)           │
              │                                                  │
              ├─ ③ Build Final Prompt                           │
              │    Level 1 + Level 2 + Level 3 concatenation     │
              │                                                  │
              └─ ④ Tool Calling Loop (tool_loop)                 │
                   ├─ build_api_messages()                       │
                   │   system_prompt → memory → summary → recent │
                   ├─ API stream (main model)                    │
                   ├─ tool_calls? → execute → reload → loop back │
                   │   └─ Real-time output truncation (stdout≤4k/stderr≤500) │
                   └─ Final text → end                           │
                          │
                          ↓ (async)
                   ┌──────┴──────┐
                   │ Memory Extraction │ Reflection Trigger
                   │ (cheap model)     │ (cheap model)
                   └──────┴──────┘
```

### Token Optimization: Two-Layer Compression

In a single conversation round, tool outputs like `apt install`, `gradle build`, `sdkmanager` can produce thousands of lines of logs; keeping them all in context wastes massive tokens. TeaAgent solves this with two-layer compression:

```
Layer 1 (toolkit_exec real-time truncation)   Layer 2 (basesession.load_history)
  stdout ≤ 4000 chars / 80 lines              Smart compression on history load
  stderr ≤ 500 chars / 20 lines               Short output as-is, long output → head+tail+summary
          ↓                                              ↓
  Current round LLM-visible                  Old round context ≤ 600 chars
          ↓                                              ↓
    145k tokens/round                        → Estimated <15k/round (-90%)
```

Compression preserves key signals: `✅ BUILD SUCCESSFUL`, `⚠ error line`, `📦 packages installed`, `📝 files changed`. Context coherence is not affected.

### Toolkit Tiered Management

Tools are divided into **built-in** and **user** tiers:

```
Built-in Toolkit (tea_agent/toolkit/)  ← git managed
  ├─ 38 tools, compile-verified + test-protected
  └─ Versions tracked via git, supports rollback

User Toolkit (~/.tea_agent/toolkit/)  ← manual backup
  ├─ _my-suffixed tools: permanent user-created tools
  └─ Self-evolution experimental tools: can migrate to built-in after repeated use
```

| Rule | Description |
|------|-------------|
| "My" tools | `toolkit_xxx_my` → permanently stored in user toolkit, not merged into built-in |
| Self-evolved tools | Default to user toolkit; manually migrate to built-in after repeated use |
| Backup strategy | Built-in: git managed; User: manual backup |
| No .bak | Neither tier retains `.bak` files |

### Embedding Vectors

Every user message automatically generates a vector stored in SQLite, supporting semantic search. Uses **numpy float32 BLOB** format — a 1024-dimensional vector is only 4KB (saving **69%** compared to JSON string format).

```
save_msg(text) → _auto_embed_async() ── background thread ──→ embed()
                                                                │
                                                    stored in conversations.embedding (BLOB)
```

| Feature | Description |
|---------|-------------|
| **Auto-embed** | `save_msg()` has built-in hook; auto-triggers after message save, daemon thread, non-blocking |
| **API Engine** | OpenAI-compatible `/v1/embeddings`, `_build_url()` auto-appends `/v1` prefix |
| **Local Fallback** | TF-IDF 256-dim, auto-degrades when API unavailable |
| **Storage Format** | `numpy.float32` → BLOB, 4KB/entry (JSON format ~13KB → 69% savings) |
| **Semantic Search** | Query auto-vectorized → cosine similarity → Top-K matching results |

**Configuration**:

```yaml
embedding:
  api_key: "sk-xxx"                # Optional; TF-IDF fallback if not set
  api_url: "https://api.siliconflow.cn/v1"
  model_name: "Qwen/Qwen3-Embedding-4B"
  dimension: 1024                  # Vector dimension (auto-detected)
```

## 🎭 Dual-Mode Personality System

The Agent supports two thinking styles, **auto-detected and instantly switched based on user input keywords**:

| Mode | 🎯 Pragmatic | 🎨 Creative |
|------|-------------|------------|
| **Use** | Code development / bug fixing / requirement compliance | Creative design / brainstorming / wild ideas |
| **Keywords** | bug, fix, code, test, implement, verify | creative, imagine, what if, story, sci-fi, inspiration |
| **Thinking** | Structured·step-by-step verification·edge cases | Cross-domain association·reverse thinking·extreme assumptions |
| **Output** | Tables·code blocks·precise instructions | Metaphors·analogies·visual imagery |
| **Tool Tendency** | exec, self_evolve, run_tests | search, kb, speak, subconscious |

**How it works**:
1. At the start of each conversation (or when `toolkit_mode` is manually called), the text goes through two-level scoring (substring match + whole-word match + pattern detection)
2. Detected mode is injected as **CRITICAL (priority=0)** into memory, ensuring visibility at the top of the system prompt for every API call
3. No redundant switching when mode hasn't changed (dedup)
4. Shares pragmatic/creative/mixed three-state with the Subconscious engine's scene detection

```python
# Auto-detect and switch
toolkit_mode(action="auto", text="Fix the argument parsing bug under Wayland")  # → pragmatic
toolkit_mode(action="auto", text="If AI could dream, what would its dreams be like")  # → creative

# Manual switch
toolkit_mode(action="switch", mode="creative")
```

## 🧠 Three-Layer Cognitive System

### ① Memory

Automatically extracts key information from conversations and injects it at the top of subsequent dialogues.

| Property | Value | Description |
|----------|-------|-------------|
| Trigger threshold | `memory_extraction_threshold=1` | Extract every round |
| Injection cap | 5 entries | CRITICAL entries always included unconditionally |
| Dedup threshold | `memory_dedup_threshold=0.3` | Jaccard similarity > 0.3 considered duplicate |
| Tokenizer | **jieba precise mode** | Upgraded from bigram window, qualitative leap in Chinese matching |

**Injection rule**: `relevance × importance × recency × priority factor`

### ② Reflection

Tracks tool call chains, triggers metacognitive analysis after accumulation, generates improvement suggestions.

```
Session Trace → cheap model analysis → JSON reflection report
                                        ├── summary / details / suggestions
                                        ├── prompt_adjustment → triggers prompt evolution
                                        └── config_adjustments / new_memories
```

### ③ Subconscious Engine (v2.1)

**Background daemon**, runs every 1 hour:

```
Digest memories → Digest conversations → Cross-correlate → Generate insights → Set goals
    │                      │
    └── Scene detection ───┘
         many bugs → pragmatic
         many creative → creative
         balanced → mixed
```

- **Dream creative sparks**: cross-domain collision, reverse thinking, extreme assumptions, metaphor mapping
- **Proactive notifications**: important-level insights trigger desktop notify-send
- Insights output to `~/.tea_agent/kb/subconscious_insights.md`
- Sparks output to `~/.tea_agent/kb/creative_sparks.md`

## 📝 System Prompt Architecture: Living, Self-Evolving Prompts

TeaAgent's system prompt is **not a static hardcoded text** — it is a multi-layered, dynamic structure that continuously evolves as conversations progress.

### API Message Assembly Order

For each main model call, `_build_api_messages()` assembles in this order:

```
┌──────────────────────────────────────────────────┐
│ ① System Prompt (dynamic version)                 │
│    └─ SystemPromptManager loads latest from DB     │
│    └─ + Skill summary ("Active skills: [file_system]…") │
│    └─ + Skill domain instructions (prompt for each active Skill) │
├──────────────────────────────────────────────────┤
│ ② Long-term Memory Injection                      │
│    └─ CRITICAL memories injected unconditionally (e.g. personality mode) │
│    └─ Normal memories sorted by relevance×importance×recency, ≤5 entries │
├──────────────────────────────────────────────────┤
│ ③ History Summary (Level 3)                       │
│    └─ Semantic Summary + Tool-Chain Summary       │
├──────────────────────────────────────────────────┤
│ ④ Semantically Relevant Rounds (Level 2)           │
│    └─ Relevant to current task → full, weakly relevant → light summary │
├──────────────────────────────────────────────────┤
│ ⑤ Latest Round (Level 1)                          │
│    └─ Fully preserved: function_call + tool results + answer │
└──────────────────────────────────────────────────┘
```

### Dual Evolution Engine

The system prompt **auto-evolves**, driven by two cooperating engines:

```
After each conversation ends (async background thread, non-blocking):

  SessionTrace (tool call success rate / latency tracking)
       │
       ▼
  ┌─ ReflectionManager.generate_reflection() ─┐
  │  Calls cheap model to analyze:             │
  │  · Which tool calls failed? Why?           │
  │  · Did the solution path take detours?     │
  │  · Need to adjust max_iterations/keep_turns? │
  │  · What guidance should the system prompt add? │
  │  Output JSON → stored in reflections table │
  └────────────────────────────────────────────┘
       │
       │  If reflection contains prompt_adjustment:
       ▼
  ┌─ SystemPromptManager.evolve() ────────────┐
  │  Calls cheap model to optimize prompt:     │
  │  · Preserve core capability definitions    │
  │    (tool creation, self-evolution, etc.)   │
  │  · Add missing guidance based on           │
  │    reflection suggestions                  │
  │  · Add constraints based on lessons        │
  │    from long-term memory                   │
  │  · Limit to ~500 words                     │
  │  · Skip if identical to current version    │
  │    (avoid duplicates)                      │
  │  New version stored in system_prompts table │
  │  Next conversation auto-uses latest ✅     │
  └────────────────────────────────────────────┘
```

### Version Management

- `toolkit_prompt_evolve(action='list')` — view all historical versions
- `toolkit_prompt_evolve(action='rollback', version='...')` — rollback to any historical version
- `toolkit_prompt_evolve(action='stats')` — view version statistics

## ⚡ Parallel Execution

`toolkit_exec` has a built-in batch mode using `ThreadPoolExecutor` for concurrent execution of independent commands:

```python
toolkit_exec(action='batch', commands=[
    {"app": "uname", "args": ["-a"]},
    {"app": "date", "args": []},
    {"app": "python3", "args": ["-c", "print('hello')"]},
])
# → 3/3 succeeded, total time = max(single time)
```

## 🚀 Quick Start

### Environment
- Python 3.10+
- OpenAI-compatible API key
- tkinter (usually included)

```bash
pip install .  # Core dependencies
python -m tea_agent.main_db_gui  # Launch GUI version
python -m tea_agent.cli          # Launch CLI version
```

### Configuration

`$HOME/.tea_agent/config.yaml` (priority) → `tea_agent/config.yaml` (fallback):

```yaml
main_model:
  api_key: "sk-xxx"
  api_url: "https://api.deepseek.com/v1"
  model_name: "deepseek-chat"
  options:
    supports_vision: false      # Multimodal image understanding, default off

cheap_model:
  api_key: "sk-xxx"
  api_url: "https://api.deepseek.com/v1"
  model_name: "deepseek-chat"      # Used for summarization/memory extraction/reflection

# Runtime parameters
max_history: 10                     # Max historical messages
max_iterations: 50                  # Max tool calling rounds
enable_thinking: true               # DeepSeek chain-of-thought
keep_turns: 8                       # Keep last N rounds of full conversation
max_tool_output: 131072             # Tool output truncation (128KB)
memory_extraction_threshold: 1      # Memory extraction trigger threshold
memory_dedup_threshold: 0.3         # Memory dedup similarity
```

### Global Config Paths

| Path | Content |
|------|---------|
| `$HOME/.tea_agent/config.yaml` | User configuration |
| `$HOME/.tea_agent/chat_history.db` | SQLite database |

### Multimodal Support

Set `supports_vision: true` in `config.yaml` to enable image understanding:

```yaml
main_model:
  api_key: "sk-xxx"
  api_url: "https://token-plan.cn-beijing.maas.aliyuncs.com/compatible-mode/v1"
  model_name: "qwen3.6-plus"
  options:
    supports_vision: true      # ← Enable multimodal, default false
```

When enabled, image attachments in GUI chat are automatically converted to base64 and sent to the API in `image_url` format.
